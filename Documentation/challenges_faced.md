## Used Gemini for Discussion

## 1. The "Server Bottleneck" Challenge
* **The Problem:** Traditional file uploads send the entire file through the Node.js backend. This consumes massive CPU/RAM and can crash the server if multiple users upload large files simultaneously.
* **Our Discussion:** We moved away from the "Proxy" model to the **Client-Side Direct Upload** pattern.
* **The Resolution:** The backend now only generates a **GCP/Supabase Signed URL**. The frontend handles the heavy lifting by streaming the binary data directly to cloud storage, making the backend infinitely more scalable.

## 2. The "Incomplete Upload" (Zombie Records) Challenge
* **The Problem:** What happens if the backend creates a database record, but the user closes their browser before the upload to GCP finishes? You end up with "zombie" data in your DB and no file in storage.
* **Our Discussion:** We implemented a **Two-Step Handshake** using a `status` flag.
* **The Resolution:** Records are created with a `PENDING` status. The frontend must call a `/confirm` endpoint after a successful upload to flip the status to `ACTIVE`. 
    * *Senior Bonus:* Mentioning a "Cleanup Cron Job" to delete `PENDING` records older than 24 hours.



## 3. The "CORS & Security" Challenge
* **The Problem:** Browsers block direct uploads to different domains (like GCP) for security. Additionally, making a bucket "Public" is a major security risk.
* **Our Discussion:** We discussed why the GCP Console UI doesn't have a CORS button and how to handle permissions without "leaking" credentials.
* **The Resolution:** 1.  We used the **GCP CLI/Cloud Shell** to apply a strict `cors.json` policy.
    2.  We kept the bucket **Private** and used **V4 Signed URLs** that expire in 5–15 minutes, ensuring temporary, secure access.

## 4. The "NoSQL vs. Relational" Challenge
* **The Problem:** Using MongoDB is often the "default" for Node.js devs, but file metadata is highly structured and relational (e.g., Files belonging to Users).
* **Our Discussion:** We debated why **PostgreSQL** is superior for this specific use case.
* **The Resolution:** We chose PostgreSQL (via **Sequelize/Prisma**) because it offers better **ACID compliance** for metadata integrity and superior **indexing** (B-Tree/GIN) for filtering files by type, size, or name.

## 5. The "Authentication Complexity" Challenge
* **The Problem:** Building a custom auth system (passwords, hashing, sessions) is "messy" and prone to security vulnerabilities.
* **Our Discussion:** We looked at how to integrate **Supabase Auth** without making the backend logic cluttered.
* **The Resolution:** We decoupled identity from business logic.
    * **Frontend:** Obtains a JWT from Supabase.
    * **Backend:** Uses a clean **Middleware** to verify the JWT. 
    * This keeps the code DRY (Don't Repeat Yourself) and highly secure.
